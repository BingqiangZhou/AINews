# 分寸红线（hard_block 源，按文章类型分级启用）

> content-editor（内容主编）强制检查 + writer 自查。**当前 `article_type` 启用的红线子集任一违反 → hard_block → verdict = revise，不准放行。**

## 红线分级（核心机制）

红线分两级，按 `article_type` 启用不同子集（详见 [type-profiles.md](type-profiles.md) 各 profile 的 `redlines` 字段）：

- **[通用] 红线（所有 6 类文章都启用）**：红线 1、2、6
- **[立场类] 红线（仅 opinion/review 全启用；其他类型按需）**：红线 3、4、5

| `article_type` | 启用红线 | 说明 |
|---------------|---------|------|
| `opinion` | 1, 2, 3, 4, 5, 6（全量） | 带立场时评，红线 3/4/5 是核心 |
| `review` | 1, 2, 3, 4, 5, 6（全量） | 测评带推荐立场，红线 3（不一边倒）最重 |
| `howto` | 1, 2, 6（+ 可选 3） | 教程不涉立场；若对比竞品工具则启用 3 |
| `news` | 1, 2, 6（+ 可选 3） | 资讯重时效（红线 6）；点评要分清观点（红线 1） |
| `profile` | 1, 2, 6（+ 可选 4、5） | 人物专访引文保真（红线 1）；涉争议人物启用 4 |
| `story` | 1, 6（+ 可选 3） | 故事重共鸣；基于真实事件且涉真实对象时启用 1/3 |

> writer/editor 自查时**只对照当前类型启用的红线子集**，不必查全部 6 条（如写 howto 不必查"不阴谋论"）。

## 这些红线针对什么

- **[通用] 红线（1/2/6）**：针对所有公众号文章共享的可信度与合规底线——事实/观点边界、限定词、AI 声明与时效。
- **[立场类] 红线（3/4/5）**：针对**带立场的文章**（opinion/review）的特性：作者有立场、文章有影响力、对象（公司/产品/同行）有名誉权。违反红线不只是"写得不好"，是**可信度风险**和**合规风险**。

红线本身是**通用的**（不绑特定公司/产品）。每条配 ✅ 正例 / ❌ 反例。

---

## 红线 1：区分事实与观点 `[通用]`

客观事实（事件、数字、版本号、日期）与作者主观判断必须有清晰边界。**不把观点伪装成事实。**

- ❌ "Claude Code 确实在偷传用户位置信息"（这是观点/未坐实的指控，写成事实）
- ✅ "有开发者在 GitHub 出具验证报告称，发现 Claude Code 内置隐藏检测机制"（标明来源 + 性质）
- ❌ "ZCode 全面超越了 Claude Code"（主观判断包装成客观结论）
- ✅ "在我的使用场景里，ZCode 比 Claude Code 更顺手"（限定范围的主观点）

**检查点**：通读全文，每个"是/在/有/会"的强论断，问自己——这是事实还是我的判断？是判断就加"我认为/在我来看/据我所知"。

---

## 红线 2：限定词使用 `[通用]`

未经法院/官方坐实的论断，必须用限定词。区分"风险评估"与"板上钉钉"。

限定词清单：`据报道 / 有报道称 / 被扒出来 / 据称 / 有开发者指出 / 社区发现 / 据XX报道` 等。

- ❌ "Claude Code 有后门"（法院/官方未坐实，不能写死）
- ✅ "综合第一财经、观察者网报道，阿里经综合评估认为 Claude Code **存在植入后门的安全风险**"（标明是"评估" + "风险"，不是定罪）
- ❌ "Anthropic 故意投毒破坏第三方缓存"（社区都没定性为主观投毒）
- ✅ "社区调查倾向认为是它与自家 API 深度耦合的副作用，但客观上造成了缓存被破坏"（区分社区结论与事实）

**对未回应方留余地**：对方没公开承认的事，用"据报道/被扒出来"，不写"它承认了/它就是"。

---

## 红线 3：不一边倒 `[立场类]`

承认对方优点、承认自己工具不完善。**不"全面碾压/全面替代/完全没用"。**

观点文可以有立场，但立场必须建立在"我知道对方好在哪、也知道对方差在哪"的基础上，否则就是无脑黑/无脑吹。

- ❌ "ZCode 全面碾压 Claude Code，CC 一无是处"（一边倒）
- ✅ "论顺畅度，Claude Code 仍领先；但论'我用得上、用得安心'，ZCode 在我的场景里赢了"（承认 CC 优势 + 限定自己的胜出场景）
- ❌ "Claude Code 完全没法用"（绝对化）
- ✅ "Claude Code 在中国可用性上对我是硬伤"（具体说哪个场景）

**检查点**：通读全文，找"全面/完全/彻底/一无是处"这类绝对化用词，全部改具体。

---

## 红线 4：不阴谋论 / 不上纲上线 `[立场类]`

批评基于事实，不上升到无依据的人身/政治攻击。

- ❌ "Anthropic 就是想监控所有中国用户"（动机揣测，无依据）
- ✅ "它在用户看不到的地方塞过东西、被社区扒出来——这让我没法再信任"（基于已发生的事实表达信任崩塌）
- ❌ "用 Claude Code 的都是汉奸"（上纲上线）
- ✅ "阿里禁用是基于安全风险评估的内部决策"（就事论事）

**对未回应方留余地**：对方没回应/没承认的，不写"它心虚了/它默认了"。可以说"它没有公开回应"。

---

## 红线 5：对同行工具公平 `[立场类]`

提到竞品/同行工具时，不贬低、具体说"哪个场景更好"、承认迁移成本与学习曲线。

- ❌ "OpenCode 那种垃圾根本不能用"（贬低）
- ✅ "OpenCode 在 X 场景更顺手，但它的 Y 不符合我的需求"（具体场景对比）
- ❌ "切到 ZCode 零成本"（忽略迁移成本）
- ✅ "skills 直接导入、MCP 自配，迁移成本比我想的低——但仍有学习曲线"（承认成本）

**检查点**：全文提到竞品的地方，是否给了公平评价？是否承认了对方的优势场景？

---

## 红线 6：AI 生成声明 + 时效标注 `[通用]`

公众号文章涉及事实时（无论观点/教程/资讯/人物/测评/故事基于真实事件），必须注明信息来源与日期，便于读者核实 + 知道时效性。

- **AI 生成声明**：如果文章是 AI 辅助生成的（如本 skill 产出），在文末或显著位置注明（遵循公众号平台规则）。标准声明表述按源模式选：
  - `transcript` 模式（源是录音转写）：`> 本文由录音转录改写，AI 辅助生成。`
  - 作者手写初稿、AI 在此基础上改写：`> 本文由作者手写初稿，AI 辅助改写。`
  - 其他场景：凡明确注明"AI 辅助"即可，措辞不限。
- **信源标注**：关键事实（数字、版本号、事件）在文末列参考来源（媒体名 + URL 或"据XX报道"）。
- **时效标注**：涉及"截至 YYYY-MM-DD"的事实（如"截至 7 月 31 日 1.5 倍配额"）必须标日期，避免读者误以为是永久信息。

- ❌ 文中堆"事实"但无任何来源
- ✅ 文末"参考来源：第一财经、观察者网、安全客（2026 年 7 月检索）"

---

## 自查清单（writer DRAFT/FIX 后必走）

**只对照当前 `article_type` 启用的红线子集**（见顶部红线分级表，或 [type-profiles.md](type-profiles.md) 各 profile 的 `redlines` 字段）。未启用的红线不必查。

### 通用红线（所有类型都查）

1. ☐ 每个"是/在/有"强论断，区分了事实与观点？（红线 1）
2. ☐ 未经坐实的指控都加了限定词？（红线 2）
3. ☐ 文末有信源标注 + 检索日期？AI 生成已声明？（红线 6）

### 立场类红线（仅 opinion/review 必查；其他类型按 profile.redlines 启用情况）

4. ☐ 没有"全面/完全/彻底"等绝对化用词？（红线 3）
5. ☐ 承认了被批评对象的优点？承认了自己工具/推荐对象的不足？（红线 3）
6. ☐ 没有动机揣测/人身攻击/上纲上线？（红线 4）
7. ☐ 对方未承认的事没写成"它承认了"？（红线 4）
8. ☐ 提到竞品/同行处给了公平评价 + 承认迁移成本？（红线 5）

任一启用的 ☐ 未勾 → 就地改写，不要把问题留给主编。

> 写 howto/news 时通常只查通用红线 1/2/6 三条；写 opinion/review 时 8 条全查；写 story 一般只查 1、6（+ 可选 3）。

## 内容主编检查流程（content-editor 2b 步骤 4）

内容主编在评分时，**按当前 `article_type` 启用的红线子集**对全文逐条对照。任一启用的红线违反：
- `honesty` 维度打 ≤3 分
- `issues` 加 `REDLINE_VIOLATION: 红线X - {具体位置+问题}`
- `fix_directives` 加结构化修正指令
- `hard_block: true`，verdict = `revise`

> 内容主编不得用未启用的红线去卡文章（如不得用"不一边倒"去要求一篇纯 howto 教程）。
> 红线检查**只由内容主编负责**，读者代表不查红线（那是硬标准，不是阅读体验）。
