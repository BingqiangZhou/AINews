## Prompt 文件格式

每张插图的 prompt 文件：`prompts/NN-{type}-{slug}.md`

需要中文文字的类型（`infographic`/`comparison`/`flowchart`/`framework`/`timeline`）必须在 frontmatter 记录图内要画的**具体中文字**；`scene` 类型不需要：

```yaml
---
illustration_id: 01
type: infographic
style: sketch-notes
palette: macaron
aspect: "16:9"
title_text: "三个被低估的副业"            # 图内主标题（文章真实中文词，非占位符）
labels: ["写作变现", "视频剪辑", "AI 配音"]  # 图内标签/节点/数据（文章真实中文词数组）
---

[Prompt content]
```

## 通用要求（所有 prompt 都包含）

```
简洁构图，留白充足。简洁或无背景。主体元素居中或按内容需要布局。
人物形象：简化的风格化剪影或象征性表现，不要写实照片风格。
所有文字使用中文，手写风格字体。只放关键词和核心概念。
```

## 颜色规范

颜色规范见 [palette-colors.md](palette-colors.md) 末尾的「颜色渲染硬约束」（权威源）：正文只用描述性颜色名、禁 hex、hex 仅入 frontmatter、末尾加「不要在图片中显示任何颜色代码、十六进制数值或色号」句。

## 文字规范

### 需要中文文字的插图类型

`infographic`、`comparison`、`flowchart`、`framework`、`timeline` 类型的插图**必须**包含中文文字：

- 所有文字使用**中文**，手写风格字体，大而醒目，语言与文章一致；只放关键词和核心概念
- **【硬约束】把要画的具体中文字固化进 frontmatter**：
  - `title_text`：图内主标题（文章真实词，非占位符）
  - `labels`：图内标签 / 节点 / 步骤 / 数据（文章真实词数组）
  - 值必须来自文章真实出现的中文词；**禁止占位符**（`[xxx]`/`{xxx}`/`{标题}`）和**禁止抽象词**（"核心概念""数据点""步骤名""关键词""方案 A"）
  - prompt 正文（visual_prompt）必须使用与 `title_text`/`labels` **相同的具体字**——后端只渲染正文里出现的文字
- **从文章中提取实际数据、术语、关键词**作为图内标签
- **禁止**在 prompt 中写"图内不要出现任何文字/中文文字/数字"之类的排除指令
- 这些类型的 prompt 结尾只需"不要在图片中显示任何颜色代码、十六进制数值或色号"

> **为什么硬约束**：图内有没有中文，取决于 prompt 正文里有没有"要画的具体中文字"。正文只有 `[数据点]`、`核心概念` 这类占位符/抽象词时，模型无字可画，结果就是无中文。把具体字同时写进 frontmatter 和正文，才能被验证、被渲染。
>
> **⚠️ 双重作用**：`title_text`/`labels` 还被 `build_segments --prompts-dir` 提取到 segments.json 的 `illustration_meta` 字段，传递给下游播客 conductor（让它知道图里画了什么，据此对齐播客要点）和覆盖校验脚本（检查播客是否讲全了 labels 里的每个词）。frontmatter 缺这两个字段 = conductor 拿不到图信息 = 播客和图语义错位。

### 不需要中文文字的插图类型

`scene` 类型的插图以视觉氛围为主：

- 可以在 prompt 末尾加"图内不要出现任何文字、字母或数字"
- 如果场景中有自然出现的文字元素（书名、屏幕内容、路标等），可以保留少量文字
- 不强制要求或排除文字，以视觉效果优先

## 片头片尾图（opening/ending）的 prompt 规范

片头图和片尾图与正文章节图一样，**必须产出 prompt 文件**到 `prompts/` 目录（文件名 `NN-opening-<slug>.md` / `NN-ending-<slug>.md`）。这与其他插图统一——`build_segments --prompts-dir` 靠 prompt 文件提取 illustration_meta。

| 图 | 基础 | type | title_text | labels |
|----|------|------|-----------|--------|
| **片头（opening）** | Step 2a 的全文 thesis（主旨/标题） | 通常 `scene`（品牌视觉总览）或 `infographic` | 文章主标题/品牌名 | 核心关键词（可选） |
| **片尾（ending）** | Step 2a 的全文 CTA（号召/总结） | 通常 `scene`（收尾氛围）或 `infographic` | CTA 文案（如"关注收藏转发"） | CTA 要素（可选） |

片头片尾图的 frontmatter 同样遵循上面的 type 规则：
- 选 `scene` 类型 → title_text/labels 可省（氛围图）
- 选 `infographic` 类型 → 必须含 title_text/labels（与正文章节图同硬约束）

outline.md 里片头片尾图的 `**Segment**` 字段填 `OPENING` / `ENDING`（不是数字），`build_segments --from-outline` 据此识别它们为 opening/ending 段。

## 原则

1. **布局结构先行**：描述构图、区域、流向
2. **具体数据/标签**：使用文章中的实际数字、术语
3. **视觉关系**：元素如何连接
4. **语义颜色**：含义驱动的颜色选择（红色=警示，绿色=高效）
5. **风格特征**：线条处理、纹理、氛围
6. **纵横比**：以比例和复杂度级别结尾

## 维度加载顺序（拼 prompt 前必读）

每张图的 prompt 由三维组合而成（Type × Style × Palette），拼装时按以下顺序加载素材：

1. **Type 骨架** → 本文件下文的「Type-Specific 模板」对应章节（决定信息结构：分区/步骤/节点/事件/场景）。Type 是 11 值单一枚举，见 [dimensions.md](dimensions.md#type11-值)。
2. **Style 渲染细节** → `styles/<style>.md`（标准六段：Design Aesthetic / Background / Color Palette / Visual Elements / Style Rules / Best For；`sketch-notes`/`ink-notes` 另含 Layout Guidelines）—— 这是 style 的视觉语言来源，把 Visual Elements / Style Rules 段的关键描述词写入 prompt 正文。
3. **派生 Rendering 风格词** → 由 Style 派生，查 [style-rendering-mapping.md](style-rendering-mapping.md) 得 rendering（7 值），从 [dimensions.md](dimensions.md#派生维度不暴露给用户) 的 Rendering 描述取 prompt cues（如 hand-drawn → "varied line weight, visible stroke texture"）。Rendering 不写入 frontmatter 作为用户维度，但可记录为派生元数据。
4. **Palette 配色（锁 warm）** → [palette-colors.md](palette-colors.md) 的 warm 段。颜色描述词写进 prompt 正文（描述性名称，禁 hex）。全局锁 warm。
5. **派生 Text/Font** → 由 Type 派生，查 [visual-preset-mapping.md](visual-preset-mapping.md) 的「Type → Text/Font 默认值」（按 Type 查，如 framework → text=text-rich, font=handwritten）。
6. **Mood 情绪** → 默认 `balanced`，不暴露。如需描述可加"harmonious contrast, balanced composition"等提示词。

> 关键：style 的精细视觉差异（线条质感、版式、装饰元素）**由 `styles/<style>.md` 的 Visual Elements / Style Rules 段写入 prompt 正文**，而 rendering 只是 7 值的渲染技法粗分（派生自 style，作为 image-generator 的 technical hint）。三维（Type + Style + Palette）+ prompt 正文细节 = 完整视觉控制。

## 插图 Text/Font 映射表

illustrate 模式生成插图时，Text 和 Font 由 Type 派生（不是用户维度）。宝玉原始方法论中文字直接嵌入 prompt 模板（LABELS、STEPS、TITLE 等），无独立 Text 维度；为避免 text=none 导致 prompt 中的文字描述不被渲染，按 Type 派生时必须显式设置。

**映射表单一信息源**：见 `visual-preset-mapping.md` → `Type → Text/Font 默认值`（按 11 值 Type 查，cover 和 illustrate 共用）。本文件不再内联该表，以避免多头漂移。

## Type-Specific 模板

> **占位符填写规则**：模板里的 `[...]`、`{标题}`、`{XXX}` 都是"用文章真实内容替换"的填空指引，**不得原样保留**。替换后的值必须是文章里真实出现的具体中文词（标题、术语、数字、步骤名），并同步写入 frontmatter 的 `title_text`/`labels`。保留方括号或填抽象词 = 失败。

```
{标题} — 数据可视化

布局：[网格/放射/层级]
区域：
- 区域 1：[数据点及具体数值]
- 区域 2：[对比与指标]
- 区域 3：[总结/结论]

```

## Type × Style × Palette 组合范例

> 以下是已组合好的 prompt 片段参照，供 agent 拼 prompt 时参考。**注意**：这些是「风格 + 配色」的视觉描述段，需配合上文 Type 骨架模板使用（骨架提供结构，本段提供外观）。完整 style 细节始终以 `styles/<style>.md` 为准。

### Comparison + ink-notes + mono-ink（前后对比 / 传统 vs 新）

```
专业手绘笔记对比图，纯白背景。黑色墨线带轻微抖动，Mike Rohde 风格。
配色：单色墨线 — 黑色墨线为主，少量语义强调色

底部：单行手写要点总结。
风格：手写标题（加粗、超大号），手写注释，充足留白，不要电脑字体，不要渐变不要阴影。

比例：16:9
```

风格：色块不填满轮廓，手写字体，充足留白。
不要在图片中显示任何颜色代码、十六进制数值或色号。
比例：16:9

## 避免的错误

- 模糊描述（"一张好看的图片"）
- 隐喻的字面插图
- 缺少具体标签/注释
- 通用的装饰元素
- 颜色名称作为可见文字出现在图片中
- 使用英文文字（所有图片内文字必须是中文）
- **对 infographic/comparison/flowchart/framework/timeline 类型写"图内不要出现任何文字"**（这些类型必须包含中文文字标签）
