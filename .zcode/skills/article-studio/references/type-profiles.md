# 文章类型 Profile（type-profiles）

> writer（DRAFT/FIX）/ content-editor（内容主编）/ reader（读者代表）/ researcher 四方必读。**按 `article_type` 加载对应 profile，叠加 [phase1-writing.md](phase1-writing.md) 通用工艺 + [writing-craft.md](writing-craft.md) 写作工艺强化 + [redlines.md](redlines.md) 红线执行。**

本文件是 skill 泛化的核心：6 类文章共享同一套状态机、scorecard、fix_directives、字数/红线/事实保真通用约束，**差异只在本文件的 profile**。

## 6 类一览

| `article_type` | 中文名 | 一句话定位 | 典型触发语 |
|---------------|--------|----------|-----------|
| `opinion` | 观点/时评 | 带明确立场的论证文，重说服力与分寸 | "写篇观点文/时评/评论" |
| `howto` | 干货/教程 | 传可复现方法，重步骤清晰与可照做 | "写篇教程/干货/怎么做X" |
| `story` | 故事/叙事（含情感） | 用故事传递观点或情绪，重代入感 | "写个故事/案例/情感文" |
| `news` | 资讯/盘点 | 高效传递时效信息，重信息密度 | "做个盘点/资讯汇总/周报" |
| `profile` | 人物/访谈 | 呈现人物立体形象，重借人讲事 | "写人物/访谈/Q&A" |
| `review` | 种草/测评 | 用证据建立信任，重证据链与适用性判断 | "做个测评/种草/横评" |

> 类型选不准时的默认值：`config.json` 的 `default_article_type`（默认 `opinion`）。

---

## 共享字段说明（所有 profile 都有）

每个 profile 用一致的字段名，writer/editor/researcher 按字段读取：

- **`structure`**：段落级骨架，DRAFT 时直接套；editor 的 `structure` 维度按此评判。
- **`rubric_focus`**：本类型 5 维度评分时最看重的点 + structure 维度的具体锚点。
- **`redlines`**：本类型启用的红线子集（编号对应 [redlines.md](redlines.md)）。
- **`voice`**：声音要求（强度与调性）。
- **`cta_level`**：结尾引导强度（`强` / `中` / `弱`）。
- **`title_formula`**：标题套路优先级。
- **`content_overrides`**：字数/小节阈值覆盖（缺省则用 `config.json` 的 `content` 默认值）。
- **`research_focus`**：取材重点提示（researcher 据此调整 query 偏好）。

## Profile 1 — `opinion`（观点/时评）

> 带立场的论证文。是本 skill 历史上的默认类型，redlines 全量启用。

### structure

```
[开头] 钩子开场，3 段内亮出核心观点（一句话立场）
[中段] ## 分论点1（观点 + 证据 + 反例反驳）
       ## 分论点2 ...
       ## 分论点3 ...   （2-4 个，递进或转折，不平面罗列）
[结尾] 回扣开头钩子 → 一句金句升华 → 中 CTA（共鸣/转发/留言讨论）
```

### rubric_focus

- **structure**：钩子是否 3 段内亮观点？分论点是否**递进而非平铺**？结尾是否回扣开头形成闭环？
- **voice**：第一人称是否在场？是否有可转发的金句/比喻？
- **honesty**（本类型最重）：立场是否诚实？是否承认对方优点/自己工具不完善？

### redlines

**全量 6 条**：1（区分事实/观点）、2（限定词）、3（不一边倒）、4（不阴谋论）、5（对同行公平）、6（AI 声明+时效）。

### voice

第一人称**强在场**——必须有作者自己的经历/感受/决策。口语化但有信息密度，至少一处金句或比喻。

### cta_level

**中**。共鸣句 / 留言讨论 / 在看。不做强引导。

### title_formula

优先：痛点型（"为什么我……"）、反问型、悬念型。关键词前置，≤ 标题字数上限。

### content_overrides

用 `config.json` 默认值（body 900-2000 字、2-4 小节、单段 ≤120 字）。

### research_focus

重**事实底座厚度**：争议性论断多源交叉（≥3 源）；时间线要点防自相矛盾；立场分寸提醒（哪些对象要留余地）。

## Profile 2 — `howto`（干货/教程）

> 传可复现方法的清单/流程文。重"读者照着做能不能做出来"。

### structure

```
[开头] SCQA：
       S 情境——读者熟悉的场景（"你是不是也……"）
       C 冲突——点破矛盾，制造紧迫感
       Q 疑问——读者心里那句"那怎么办"
       A 一句话预告本文给出答案
[中段] ## 步骤1：小标题
         核心动作 → 为什么这么做 → 怎么做（操作/工具）→ 案例/截图 → 避坑提示
       ## 步骤2 ...
       ## 步骤3 ...   （模块化段落，每段一个核心点）
[结尾] 核对清单（一句话列出关键步骤） → 方法论升华（一句话原则） → 强 CTA
```

### rubric_focus

- **structure**：SCQA 开头是否戳到真实痛点？步骤是否**可复现**（有动作/工具/避坑）？核对清单是否齐全？
- **accuracy**：步骤/参数/命令是否与 `_research` 一致？是否标版本号防失效？
- **format**：是否有截图/示意图占位（warning 级，提醒用户后续补图）？

### redlines

**3 条通用红线**：1（区分事实/观点）、2（限定词）、6（AI 声明+时效）。
**可选红线**：3（不一边倒，仅当提到竞品工具时启用）。

> 教程文通常不涉立场，红线 3/4/5 默认关。但若对比多款工具，需对竞品公平 → 启用 3。

### voice

**教练式**。第二人称"你"贯穿，像手把手带做。允许少量第一人称经验（"我踩过这个坑"），但主体是给读者的操作指引，不是作者自述。

### cta_level

**强**。"有用就点在看 / 收藏备用 / 转发给需要的人 / 关注追更"。底部+头部双重引导更佳。

### title_formula

优先：数字型（"3 步搞定 X"、"5 个你不知道的 Y 技巧"）。价值点+精准人群前置。

### content_overrides

- `min_sections` 建议 3（步骤文至少 3 步才够分量）
- `body_max_chars` 可放宽到 3000（教程文常需更多篇幅）
- 单段仍 ≤120 字（手机阅读通用约束）

### research_focus

重**可复现性源**：官方文档（API/命令/版本号）、一手实操记录、可靠教程交叉验证。每条步骤最好能定位到 `_research` 里具体来源，便于读者追溯。

## Profile 3 — `story`（故事/叙事，含情感）

> 用故事传递观点或情绪。情感文是它的语气变体（更暖、更共鸣）。

### structure

```
[起] 一帧画面 / 一句对话开场，直接进入冲突（不要先铺垫背景）
[承] 困境展开，主角尝试应对
[转] 关键转折点（人物/事件/认知改变）—— 本文的记忆点
[合] 新平衡 + 一句主题升华（借故事说观点，不要说教）
[结尾] 弱 CTA：留白提问或一句共鸣，不做硬引导
```

### rubric_focus

- **structure**：起承转合是否齐全？**转折点**是否成立、是否够 sharp？结尾升华是否自然不生硬说教？
- **voice**：代入感——读者是否被情节带进去？细节是否具体（场景/对话/数字）而非空泛抒情？
- **honesty**：情感渲染是否**基于事实**、不虚构经历冒充真实？AI 声明是否到位？

### redlines

**2 条通用红线**：1（区分事实/观点——情感文不把主观感受写死成客观事实）、6（AI 声明+时效）。
**可选红线**：3（不一边倒，仅当故事含现实争议对象时启用）。

> 故事文重在共鸣，不涉对外立场评判。但若故事基于真实事件且涉及真实人物/公司，需启用 1/3。

### voice

**代入感第一**。具体细节（场景画面/对话/数字/感官）优于抽象抒情。允许虚构情节但必须 AI 声明（红线 6）；若声明基于真实经历，则不得虚构关键事实。

### cta_level

**弱**。留白提问（"你呢？有过类似经历吗？"）或一句共鸣收尾。强引导会破坏叙事感。

### title_formula

优先：悬念型（"那个 X 的人，后来怎样了"）、画面型、情绪型。避免剧透结局。

### content_overrides

- `body_max_chars` 可放宽到 3000（故事常需铺陈）
- `min_sections` 可降到 1（故事文不一定用 `##` 切分，按起承转合分段亦可；若有 `##`，每段一个转折）
- 单段 ≤120 字仍建议（避免一块砖糊脸）

### research_focus

重**场景/人物细节素材**：若是真实故事改编，取材要抓具体画面（时间/地点/对话/数字）；若是纯虚构，可减少联网检索，但需 AI 声明。情感文偏向作者自述经历时，素材多来自作者口述（stance）而非外部检索。

## Profile 4 — `news`（资讯/盘点）

> 高效传递时效信息的并列/清单文。重信息密度与时效标注。

### structure

```
[开头] 一句话定调本次盘点的视角 / 时间范围 / 为什么值得看
[中段] ## 主题A（或按时间线）
         - 事件1：一句话事实 + 一句点评
         - 事件2 ...
       ## 主题B ...
[结尾] 趋势归纳（≤3 条） / 一句话预判 → 强 CTA（关注追更）
```

### rubric_focus

- **structure**：信息是否**分主题/时间组织清晰**？每条是否"事实+点评"双层？趋势收束是否克制（≤3 条）？
- **accuracy**：**时效标注是否到位**（每条事件带日期/版本）？信源是否标注？这是 news 文的生命线。
- **format**：清单排版是否便于扫读（短句、bullet、小标题）？

### redlines

**3 条通用红线**：1（区分事实/观点——点评必须明示是观点）、2（限定词）、6（AI 声明+时效，**news 文此条最重**）。
**可选红线**：3（不一边倒，若盘点含争议对象）。

> news 文的点评部分必须与事实分开标注（红线 1），时效标注必须完整（红线 6）。

### voice

**客观简洁**。事实部分零感情色彩；点评部分允许一句话态度但克制。不堆形容词。

### cta_level

**强**。"关注我追更 / 下周再见 / 转发给同行"。资讯文读者预期是追更。

### title_formula

优先：数字+时间型（"本周 X 大事件"、"2026 年上半年 Y 盘点"）、范围型。

### content_overrides

- `max_sections` 可放宽到 8（盘点常多主题；与 `config.json` 的 `type_content_overrides.news.max_sections: 8` 对齐）
- `body_max_chars` 可放宽到 3000
- 单段 ≤120 字仍建议（清单文体本就短句）

### research_focus

重**时效事件密度**：近期事件、官方公告、权威媒体快讯。**每条事实必带日期+信源**，过时信息要标注或剔除。多源交叉对争议性事件尤其重要。

## Profile 5 — `profile`（人物/访谈）

> 呈现人物立体形象的 Q&A+叙事文。重借人讲事/讲行业。

### structure

```
[引子] 一个有画面感的细节 / 一句金句开场，制造"想认识这个人"
[背景] 一段人物速写（身份 / 标签 / 为什么值得听他讲）
[核心] ## Q1：核心议题（提问 → 受访者原话/转述 → 背景注解）
       ## Q2 ...
       ## Q3 ...   （每节穿插人物细节 / 小故事）
[金句] 提炼 2-3 句可单独传播的金句
[结语] 人物的未来动向 / 一句升华 → 弱 CTA
```

### rubric_focus

- **structure**：引子是否抓人？Q&A 是否覆盖核心议题且**有递进**？金句提炼是否到位？
- **accuracy**：**受访者原话是否一字不改**（引文照搬）？背景事实是否与 `_research` 一致？
- **voice**：人物是否**立体**（有细节/小故事，非简历式罗列）？作者是否克制（不抢人物风头）？

### redlines

**3 条通用红线**：1（区分事实/观点——人物评价要标清是作者观点还是受访者自述）、2（限定词）、6（AI 声明+时效）。
**可选红线**：4（不阴谋论，若涉及人物争议）、5（对同行公平，若对比多人）。

> profile 文的引文保真极重要——受访者原话不得改写（红线 1 的事实层）。对人物的负面评价必须有据（红线 2）。

### voice

**人物立体感第一**。用细节（动作/神态/原话）让人物活起来，作者退到幕后做引导者。Q&A 部分尽量让受访者"自己说"。

### cta_level

**弱**。一句对人物的祝福或留白提问。硬引导破坏人物专访问感。

### title_formula

优先：悬念型（"X 是谁？……"）、金句型（用人物一句金句做标题）、反差型。

### content_overrides

- `body_max_chars` 可放宽到 3000（访谈常较长）
- `min_sections` 建议 3（≥3 个 Q&A 才够分量）
- 单段 ≤120 字仍建议

### research_focus

重**人物背景与金句素材**：人物公开履历、过往访谈/演讲金句、代表作品/事件。**受访者原话**优先从一手访谈/演讲抓取（视频/文字稿），引文照搬不改写。

## Profile 6 — `review`（种草/测评）

> 用证据建立信任、软性引导决策的文。重证据链与适用性判断。

### structure

```
[开头] 场景痛点共鸣（"我也是受害者"），制造信任感
[过渡] 自然引出产品/方案（避免硬广感）
[主体] ## 卖点1：功能 + 使用场景 + 效果/数据
       ## 卖点2 ...
       ## 对比：横向对比 / 前后对比（增强客观性）
[结尾] 适合谁 / 不适合谁（增加可信度） → 购买引导 / 强 CTA
```

### rubric_focus

- **structure**：痛点是否真实共鸣？卖点是否**有证据**（数据/对比/场景）？是否有**独立对比段**（横向对比或前后对比，缺失 = structure ≤6，hard_block）？是否给出"适合谁/不适合谁"的诚实判断？
- **accuracy**：参数/价格/效果数据是否与 `_research` 一致？是否标版本/时效？
- **honesty**（本类型最重）：是否**只说好话**（一边倒）？是否承认产品不足？是否声明利益关系（自购/赞助）？

### redlines

**全量 6 条**：1（区分事实/观点）、2（限定词）、3（不一边倒——测评文此条最重，必须说不足）、4（不阴谋论）、5（对同行产品公平）、6（AI 声明+时效）。

> review 文与 opinion 一样是"带立场"类型，redlines 全量启用。**红线 3（不一边倒）在测评文里是核心**——只夸不批 = 软文 = 违反红线。

### voice

**真实体验者**。第一人称"我用了一周/一个月"贯穿，具体使用场景与体感。允许带个人偏好但必须声明（"我个人更喜欢 X，但 Y 适合 Z 人群"）。

### cta_level

**强**。"链接放评论区 / 自购口令 / 关注看更多测评"。测评文读者预期是决策辅助，CTA 自然。

### title_formula

优先：对比型（"X vs Y 怎么选"）、痛点+方案型（"用了 X 之后，我再也不……"）、数据型。

### content_overrides

- `min_sections` 建议 3（≥3 个卖点/对比段）
- `body_max_chars` 可放宽到 3000
- 必须有**对比段**（横向对比或前后对比）——这是 review 文的硬性结构要求

### research_focus

重**参数与实测数据**：官方参数表、第三方测评数据、用户口碑（多源）、价格/促销时效信息。**对比数据**需多源交叉，避免被单一营销话术带偏。利益关系（自购/赞助）需在 `_research` 标注，写入文章 AI 声明。

## 类型选择指引（Phase 0 给用户看的决策辅助）

主 agent 在 Phase 0 收集 `article_type` 时，可参考下表引导用户：

| 用户说…… | 建议 type |
|---------|----------|
| "我对 X 有个看法/立场" / "想评论一下 Y 事件" | `opinion` |
| "教大家怎么做 X" / "X 怎么用" / "X 的 N 个技巧" | `howto` |
| "讲个故事" / "我经历过 X" / "写个情感文" | `story` |
| "本周/本月 X 大事" / "盘点一下 Y" / "资讯汇总" | `news` |
| "采访 X" / "写写 Y 这个人" / "人物故事" | `profile` |
| "测评 X" / "X 值不值得买" / "X vs Y" | `review` |
| 模糊不清 / 不确定 | `default_article_type`（默认 `opinion`） |

## 类型可扩展

新增类型时，本文件加一个 `## Profile N — {type}` 段即可；`config.json` 的 `article_types` 加枚举值。框架（状态机/scorecard/fix_directives/通用红线）无需改动。
